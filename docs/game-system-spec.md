# ゲームシステム仕様

この文書は、現在の実装から読み取れるゲームシステム仕様をまとめたものです。対象は `src/game` 配下のゲームロジック、ゲーム進行を制御する `src/app/App.tsx`、およびゲーム画面コンポーネントです。

## ゲーム概要

- タイトル: `本番中 x 舞台裏`
- タイトルコピー: `兆候を読み 予定外を名場面へ`
- ジャンル: 1人用の舞台裏マネジメントゲーム
- プレイヤーの役割: 本番中の裏方。役者へ直接命令するのではなく、兆候を読み、本番前の準備を決め、予定外の出来事へ対応する。
- 1公演の長さ: 全3日、各日マチネ/ソワレの2回、合計6ターン。
- 1ターンの基本構造:
  1. 焦点役者と兆候を確認する。
  2. 「準備」を1つ選び、本番前メモへ反映する。
  3. 役者の出来事が解決される。
  4. 本番中の出来事に対して「対応を決める」。
  5. 場面結果を確認し、スコアへ反映する。
- 勝敗条件: 明示的なゲームオーバーはない。6ターン終了時の公演結果、レビュー、ハイライトによって評価される。

## 公演日構成

| 日 | 名前 | 役割 | ターン |
| --- | --- | --- | --- |
| 1 | 1日目 | 呼吸作り。準備と座組信頼が公演の型を作る | 1-2 |
| 2 | 2日目 | 揺らぎ。型を活かして攻めるか、負荷を整える | 3-4 |
| 3 | 3日目 | 千秋楽。残った信頼と負荷を回収して閉じる | 5-6 |

日と公演回は総ターンから計算する。

- `act = ceil(totalTurn / 2)`。内部名は既存互換のため `act` のままだが、UIでは「日」として扱う。
- `turnInAct = 1` はマチネ、`turnInAct = 2` はソワレ。

各日ソワレの結果確定後、裏方負荷が1軽減される。

| 公演回 | 役割 | 結果表示 |
| --- | --- | --- |
| マチネ | 昼公演。見極めと調整が効きやすい | 次の回への申し送りと主な理由を短く表示する |
| ソワレ | 夜公演。評判は伸びるが、負荷も残りやすい | 厚め。その日の評判、公演の型への影響を表示する |

1日目ソワレの結果確定時に「公演の型」を決める。型は2日目以降の判断傾向を少し変える。

| 型 | 条件傾向 | 効果 |
| --- | --- | --- |
| 熱量で押す公演 | 拾う、評判が高い | 拾う成功で評判が伸びやすいが、負荷も残りやすい |
| 間で残す公演 | 待つ、座組信頼が高い | 待つ成功で座組信頼が伸び、準備が外れても崩れにくい |
| 段取りで支える公演 | 整える、段取りや負荷回復が高い | 整える成功で負荷が抜けやすく、ほころびを抑えやすい |
| 崩れを閉じる公演 | 切る、低負荷で閉じる | 切るで事故を抑えやすいが、評判は伸びにくい |

## 進行状態

ゲーム状態 `status` は以下の5種類。

| 状態 | 内容 |
| --- | --- |
| `title` | タイトル画面。新規開始、履歴から同じ巡り合わせで再演、遊び方表示ができる。内部的には同じseedで開始する。 |
| `prep` | 準備フェーズ。注目役者、状態、発生しやすい兆候上位3件、役者の癖、次ターン候補、準備カード、本番前メモを表示する。 |
| `response` | 対応を決めるフェーズ。準備確定後に役者イベントが確定し、4つの対応を選ぶ。 |
| `result` | 結果プレビュー表示。マチネはソワレへの影響を薄く、ソワレはその日の評判と型への影響を厚く表示する。 |
| `finished` | 終演画面。最終結果、レビュー、ハイライト、再演操作を表示する。 |

操作アクションは以下。

| アクション | 有効状態 | 効果 |
| --- | --- | --- |
| `START` | 任意 | 新しい公演を開始する。seed指定時はそのseedで開始する。 |
| `RESET_TO_TITLE` | 任意 | タイトル状態へ戻る。 |
| `SELECT_PREP` | `prep` | 準備を保存し、役者イベントを解決して `response` へ進む。UI上は本番前メモの承認欄が `未承認` から `承認済` になってから進む。 |
| `SELECT_RESPONSE` | `response` または `result` | 対応を保存し、`result` へ進む。 |
| `COMMIT_RESULT` | `result` | 結果を確定し、スコア・ログ・役者状態・次ターンを更新する。 |

## スコア

管理する公演スコアは4種類。

| スコア | 初期値 | 範囲 | 意味 |
| --- | --- | --- | --- |
| 評判 `sceneScore` | 0 | 制限なし | 場面化、名場面化の積み上げ。ターン結果でも数値スコアは「評判」として表示する。 |
| 段取り `flowScore` | 0 | 制限なし | 公演全体の進行の安定度。ターン結果では「流れ」として表示する。 |
| 座組信頼 `trustScore` | 0 | 制限なし | 役者との信頼、終盤の伸びやすさ。ターン結果では「信頼」として表示する。 |
| 裏方負荷 `backstageLoad` | 0 | 0-5 | 裏方の負担。高いほど結果判定に不利。 |

裏方負荷は `clampLoad` により常に0以上5以下に丸められる。

## 役者

役者は3人。

| ID | 表示名 | 初期状態 | 初期信頼 | 初期疲労 | 初期役割 |
| --- | --- | --- | --- | --- | --- |
| `lead` | 主役 | 沈思 | 0 | 0 | 焦点 |
| `junior` | 若手 | 高揚 | 0 | 0 | 兆候 |
| `skilled` | 技巧派 | 没入 | 0 | 0 | 待機 |

役者が持つ値。

| 値 | 内容 |
| --- | --- |
| `state` | 現在の心理・身体状態。 |
| `trust` | 個別役者の信頼値。内部的に蓄積されるが、現時点では直接スコア計算に使われない。 |
| `fatigue` | 0-3の疲労値。3以上になると状態が疲労へ固定される。 |
| `currentRole` | 画面上の役割。`focus`、`sign`、`waiting`。 |

### 役者状態

| ID | 表示名 |
| --- | --- |
| `elated` | 高揚 |
| `contemplative` | 沈思 |
| `anxious` | 不安 |
| `immersed` | 没入 |
| `fatigued` | 疲労 |

状態変化候補は `高揚 -> 沈思 -> 不安 -> 没入 -> 疲労` の配列からランダムに選ばれる。循環順に進むのではなく、この配列を候補リストとして使う。

## 焦点役者

各ターンの焦点役者はseed付き乱数で決まる。優先順は以下。

| ターン | 優先焦点 |
| --- | --- |
| 1 | 若手 |
| 2 | 主役 |
| 3 | 技巧派 |
| 4 | 若手 |
| 5 | 主役 |
| 6 | 技巧派 |
| 7 | 若手 |
| 8 | 技巧派 |
| 9 | 主役 |
| 10 | 若手 |

判定手順。

1. `seed:focus:totalTurn` から乱数を作る。
2. 72%の確率で優先焦点を採用する。
3. 28%の確率で上記の優先順リスト全体からランダムに選ぶ。

焦点以外の役者の表示役割は以下。

- 焦点役者は `focus`。
- 焦点以外で、若手または状態が不安の役者は `sign`。
- それ以外は `waiting`。

## 役者イベント

イベントは8種類。

| ID | 表示名 | 説明 |
| --- | --- | --- |
| `stepForward` | 前へ出る | 予定より半歩前に出て、客席の視線をさらいかけている。 |
| `adlib` | アドリブ | 台詞の隙間に、台本にはない一言が落ちた。 |
| `heatUp` | 熱が乗る | 言葉と身体に熱が宿り、場面が大きく膨らみ始めた。 |
| `silence` | 沈黙する | 台詞が止まり、舞台に息を呑むような間が生まれた。 |
| `positionShift` | 立ち位置がズレる | 立ち位置が少しズレ、照明と視線の軸が変わった。 |
| `tempoRush` | テンポが走る | 呼吸より先に台詞が走り、舞台全体の拍が揺れている。 |
| `delayedExit` | 退場が遅れる | 退場の一歩が遅れ、余韻と進行の境目で揺れている。 |
| `ensembleWaver` | 群像が揺れる | 周囲の動きがわずかに乱れ、群像の輪郭が揺らいだ。 |

イベント解決は以下。

1. 焦点役者の種類別重みと状態別重みを足す。
2. 焦点役者の疲労が2より大きい場合、`positionShift`、`tempoRush`、`ensembleWaver` にそれぞれ+3する。
3. `seed:event:totalTurn:focusId` から乱数を作る。
4. 基本は合計重みによる加重抽選を行う。
5. `0.14 + backstageLoad * 0.04` の確率で本番の揺れが入り、重みを平準化した抽選を行う。高負荷ほど、見えていた上位兆候から外れる余地が増える。

準備は役者イベントの抽選には影響しない。準備は「起きた出来事に備えられていたか」と「本番中の対応へどうつながるか」の判定に使う。

## 準備

準備フェーズでは4種類から1つ選ぶ。準備はイベント抽選seedには含まれず、結果スコアの準備品質判定に影響する。

| ID | 表示名 | 活きやすい対応 | 外れた時の受け方 | 備える出来事 |
| --- | --- | --- | --- | --- |
| `watch` | 注視 | 拾う | 整えるで安全寄りに処理できる | 前へ出る、アドリブ、熱が乗る |
| `makeSpace` | 余白 | 待つ | 熱が来たら拾うと伸びる | 沈黙する、退場が遅れる、熱が乗る |
| `tightenFlow` | 締め | 整える | 高負荷なら切って守る手もある | 立ち位置がズレる、テンポが走る、群像が揺れる |
| `prepareTransition` | 転換 | 切る | 低負荷なら整える・待つで場面化を狙える | テンポが走る、退場が遅れる、群像が揺れる、立ち位置がズレる |

発生イベントが選択した準備の対応イベントに含まれる場合、結果スコアに+1され、上振れ幅と副作用の抑制に効く。結果画面では `準備が活きた`、`準備が一部活きた`、`別の備えだった` といった関係を表示する。

準備カードには、準備名、備える兆候の灯り、活きやすい対応アイコン、備える揺れカテゴリ、メモ反映状態を表示する。対応表示は「次の行動が固定された」という意味ではなく、「この準備は本番でその対応を選ぶと活かしやすい」という関係を示す。文章での説明は選択中詳細や本番前メモに寄せる。

選択中の準備は本番前メモへ反映される。メモには見えている兆候、同じ準備で拾える出来事、備えどおりに来た場合、外れた場合、承認欄を表示する。CTA `この準備で本番へ` を押すと、メモ紙上の承認欄が `未承認` から `承認済` へ変わり、対応を決めるフェーズへ進む。

## 対応を決める

準備確定後、役者イベントに対して4種類から1つ選ぶ。

| ID | 表示名 | 説明 | 主な負荷系統 |
| --- | --- | --- | --- |
| `catch` | 拾う | 予定外の行動を見せ場に変える | 光 |
| `arrange` | 整える | 乱れを舞台全体の呼吸に戻す | 進行 |
| `wait` | 待つ | 間や余韻を信じて急かさない | 音 |
| `cut` | 切る | 場面を閉じ、転換と道具を次へ送る | 道具 |

対応は役者イベントとの相性、役者タイプ、役者状態、公演日/公演回、公演の型、連続使用、信頼、負荷、ほころび回収に影響する。

対応カードには、各候補について以下を短く表示する。

- 手の性格。
- 選んだ準備との関係。
- 成功幅。
- `ほころび回収`、`次へ渡す`、`安定させる` などの短い判断ラベル。
- 主要影響2件。

詳細な狙い、上振れ・下振れ、相性盤、パッシブ効果、公演回の影響、負荷・信頼・流れへの副作用は、選択中の対応だけ進行卓 HUD に表示する。

画面上では、対応カードは候補比較、進行卓 HUD は選択中の対応を送出する判断面として分ける。対応カードの準備ラベルは `準備活きる`、`備え外`、`準備合わず` の短い言葉で表示し、aria と進行卓で正式な意味を補う。`攻め手`、`安定手`、`余韻手`、`収束手` のような手の性格バッジは、対応名や狙いと情報が重なるためカード上には出さない。

進行卓 HUD には、選択中の対応、準備とのつながり、場面の見通し、判定ランプとしての相性盤、影響ゲージとしてのキュー後の影響、ほころび回収、発動中のパッシブ効果を表示する。CTA は進行卓内の情報パネルには入れず、進行卓の外にキューバーとして分離する。キューバーは対応カードと進行卓の間に置き、画面下部に追従する。進行卓本文にはキューバー分の下余白を確保し、相性盤や影響欄を隠さない。文言は `このキューを出す` に集約する。

相性盤は今後の高度な判断に必要な情報として扱うため、折りたたまない。`出来事`、`役者型`、`状態`、`公演回` の4項目を常時表示する。表示は判定ランプを主役にし、文字は `強い`、`合う`、`普通`、`注意` の短い言葉だけにする。`◎`、`○`、`△`、`×` は画面上では出さず、内部評価やアクセシビリティ用の意味として扱う。

### 画面上の判断理由

対応カードには、各候補の性格、成功幅、判断理由が出る。最終結果ランクはここでは出さず、結果画面で初めて表示する。

| 項目 | 表示例 |
| --- | --- |
| 準備との関係 | `準備が活きる`、`備え外で効く`、`準備と合わない` |
| 狙い | `予定外を見せ場に変える` |
| 成功幅 | `ほころび〜小さな成功` |
| 上振れ | `噛み合えば場面化まで届く` |
| 下振れ | `負荷が残る` |
| 相性盤 | `出来事 強い`、`役者型 合う`、`状態 合う`、`公演回 普通` |
| 副作用 | `負荷-1 / 信頼+1 / 評判+1` |

候補カードのトーンは成功幅と危険警告に応じて変わる。`事故` は通常の見込みとしては表示せず、下振れが事故の恐れを含む時だけ `危険: 事故の恐れ` として表示する。

| 状態 | トーン |
| --- | --- |
| 名場面まで伸びる | best |
| 場面化まで届く | good |
| ほころびが絡む | thin |
| 事故の恐れ警告 | danger |

## 結果スコア計算

結果プレビュー時、内部スコア `score` を以下の合計で計算する。

```txt
score =
  イベント相性
  + 役者タイプ補正
  + 役者状態補正
  + 準備品質補正
  + 公演日/公演回補正
  + 公演の型補正
  + 信頼補正
  + 負荷ペナルティ
  + 連続使用ペナルティ
  + ほころび回収補正
```

### 役者タイプ補正

| 条件 | 補正 |
| --- | ---: |
| 主役に `wait`、`catch`、`arrange` | +1 |
| 若手に `catch`、`arrange` | +1 |
| 技巧派に `wait`、`arrange` | +1 |
| 高揚状態の若手に `wait` | -1 |

### 役者状態補正

| 条件 | 補正 |
| --- | ---: |
| 高揚に `catch` | +1 |
| 沈思に `wait` | +1 |
| 不安に `arrange` | +1 |
| 没入に `wait` または `catch` | +1 |
| 疲労に `arrange` または `cut` | +1 |
| 疲労に `catch` | -1 |

### 準備品質補正

準備は3段階で評価する。

| 品質 | 条件 | 効果 |
| --- | --- | --- |
| 活きた | 発生イベントが選択した準備の対応イベントに含まれる | +1。上振れ幅が広がり、副作用を抑えやすくなる。 |
| 一部活きた | 発生イベントとは違ったが、見えている兆候には備えていた | 補正0。場面化まで安定しやすい。 |
| 別の備えだった | 発生イベントにも見えている兆候にも備えていなかった | -1。上限が小さな成功まで下がり、攻め手は負荷が残りやすい。 |

一部活きた場合の上限は場面化、別の備えだった場合の上限は小さな成功。直前に `cut` で場面を閉じていた場合は、別の備えでも上限が場面化まで広がる。準備が活きた時は基本的に上限なし。ただし `arrange` は安定手として扱い、焦点役者が技巧派、不安/疲労状態、またはほころび回収などの強い理由がない場合、準備が活きても場面化までに留まる。

### 準備と対応の関係

各準備には、本番で選ぶと活きやすい対応がある。UI表面では対応アイコンと短い関係ラベルで示し、行動固定ではなく相性として読めるようにする。内部的には外れた場合の受け方も残す。

| 準備 | 活きやすい対応 | 効き方 | 外れた時の受け方 |
| --- | --- | --- | --- |
| 注視 | 拾う | 準備が活きた時に場面の伸びを少し広げる | 整えるで安全寄りに処理できる |
| 余白 | 待つ | 信頼と余韻を守りやすい | 熱が乗る出来事なら拾う攻め筋もある |
| 締め | 整える | 乱れを流れへ戻しやすい | 高負荷時は切る守り筋もある |
| 転換 | 切る | 下振れ、ほころび、負荷悪化を抑えやすい | 低負荷なら整える・待つで場面化を狙える |

準備が活きたうえで対応も噛み合った場合、追加補正を得る。`prepareTransition + cut` は、転換系の出来事を閉じて次へ渡す切れ味として追加補正を得つつ、下振れ軽減と負荷悪化抑制も持つ。

### 公演日/公演回補正

| 条件 | 補正 |
| --- | ---: |
| 1日目で `wait` または `arrange` | +1 |
| 1日目ソワレで `catch` | +1 |
| 2日目で `catch` | +1。ソワレならさらに強い |
| 2日目で `arrange` または `cut` | +1 |
| 3日目で `wait` または `arrange`、かつ信頼スコアが2以上 | +2 |
| 3日目で `cut`、かつ負荷3以上 | +1 |
| 3日目ソワレで負荷3以上、かつ `arrange` または `cut` | +1 |
| 3日目ソワレで信頼4以上、かつ `wait` | +1 |
| 3日目ソワレで負荷1以下、かつ `catch` | +1 |

### 公演の型補正

1日目ソワレ後に決まった型は、2日目以降の対応に小さな補正を与える。

| 型 | スコア補正 | 結果差分 |
| --- | --- | --- |
| 熱量で押す公演 | `catch` +1 | `catch` 成功時、評判+1、負荷+1 |
| 間で残す公演 | `wait` +1 | `wait` 成功時、座組信頼+1 |
| 段取りで支える公演 | `arrange` +1 | `arrange` 成功時、負荷-1追加 |
| 崩れを閉じる公演 | `cut` +1 | `cut` 時、負荷悪化とほころびを抑えやすい |

### 信頼補正

```txt
trustBonus = clamp(floor(trustScore / 4), -2, 2)
```

信頼スコアが高いほど最大+2、低いほど最大-2まで結果スコアに影響する。

### 負荷ペナルティ

| 裏方負荷 | 補正 |
| ---: | ---: |
| 0-2 | 0 |
| 3 | -1 |
| 4 | -3 |
| 5 | -4 |

直前ターンで `cut` を選んで場面を閉じていた場合、次ターンの負荷ペナルティは1段階軽くなる。

### 連続使用ペナルティ

現在選んだ対応を含めて、同じ対応が何回連続しているかを `count` とする。`catch`、`arrange`、`wait` は2連続までは戦術として許容し、3連続以上で内部スコアに-1する。`cut` は単発で閉じる判断として強いぶん、2連続から内部スコアにペナルティが入る。

| 対応 | 2連続 | 3連続以上 |
| --- | --- | --- |
| `catch` | スコア補正なし | -1 |
| `arrange` | スコア補正なし | -1 |
| `wait` | スコア補正なし | -1 |
| `cut` | -1 | -3 |

直前と違う対応を選んだ場合、`count = 1` となりペナルティはない。

### ほころび回収補正

`pendingFrayEvent` があり、舞台裏のほころび系統と対応が合う場合は+1。強く合う場合は追加で+1され、結果ランクを1段階底上げし、成功時は評判・流れ・信頼にも+1される。同じ対応を直前ターンから続けている場合は、作った負荷を踏み続けている扱いになり、この追加報酬とランク底上げは発生しない。合わない場合は内部スコア加点はなく、結果差分で流れ-1、負荷4以上なら負荷+1が追加される。

| ほころび系統 | 強く拾える対応 | 整えられる対応 |
| --- | --- | --- |
| 照明 | `catch` | `arrange` |
| 音響 | `wait` | `arrange` |
| 進行 | `arrange` | `cut` |
| 道具 | `cut` | `arrange` |

内部スコアから結果ランクを決める。

| 内部スコア | ランクID | 表示名 | 星 |
| ---: | --- | --- | --- |
| 7以上 | `masterpiece` | 名場面 | ★★★★★ |
| 4-6 | `scene` | 場面化 | ★★★★☆ |
| 2-3 | `smallSuccess` | 小さな成功 | ★★★☆☆ |
| 0-1 | `fray` | ほころび | ★★☆☆☆ |
| -1以下 | `accident` | 事故 | ★☆☆☆☆ |

## 結果差分

結果ランクごとの基本差分。

| ランク | 評判 | 流れ | 信頼 |
| --- | ---: | ---: | ---: |
| 名場面 | +4 | +2 | +2 |
| 場面化 | +3 | +1 | +1 |
| 小さな成功 | +1 | +1 | 0 |
| ほころび | 0 | -1 | -1 |
| 事故 | -1 | -2 | -2 |

結果画面では差分を数字だけで出さず、以下の3点を併記する。

- 体感ラベル: `見せ場級`、`呼吸が戻った`、`重い代償` など。
- メーター: 評判、流れ、信頼は `4/4`、負荷は裏方負荷ゲージと合わせて `5/5` のように評価幅内の位置を見せる。
- 内部数値: 対戦・攻略で必要になるため `+4`、`-1` の数値も残す。

評判の `+4` は `最大` ではなく `見せ場級` と表現する。これは計算上の最大幅は示しつつ、演出的には「この場面が客席に残った」ニュアンスを優先するため。

### 負荷差分

| 対応 | 成功系ランク | 低結果系ランク |
| --- | ---: | ---: |
| `catch` | 名場面なら+2、それ以外は+1 | +2 |
| `arrange` | -1 | +1 |
| `wait` | -1 | +1 |
| `cut` | 基本±0 | 基本±0 |

成功系ランクは `masterpiece`、`scene`、`smallSuccess`。低結果系ランクは `fray`、`accident`。

追加で以下がかかる。

| 条件 | 効果 |
| --- | --- |
| 2日目で `catch` | 負荷+1。揺らぎの日では攻めの代償が大きい。 |
| 3日目で `arrange` または `cut` が成功 | 負荷-1。千秋楽では立て直しが効く。 |
| マチネで `arrange` または `wait` が成功 | 負荷-1追加。昼公演は調整が効きやすい。 |
| ソワレで `catch` が成功 | 負荷+1追加。夜公演は評判が伸びるぶん負荷が残る。 |
| 別の備えだった状態で `catch` | 負荷+1。無理に拾うと負荷が残る。 |
| 準備が活きた状態 | 負荷増加がある場合、+1分を抑制する。 |
| `prepareTransition` で `cut` | 負荷増加がある場合、+1分を抑制する。 |
| 直前ターンで `cut` | 負荷増加がある場合、+1分を抑制する。 |

`wait` 成功時は信頼+1。`cut` 成功時は通常は信頼-1だが、高負荷局面または `prepareTransition + cut` では、閉じる必要がある判断として扱い信頼低下を抑える。負荷上昇は低結果だけでなく、攻めの代償として扱う。

### 連続使用による結果差分補正

同じ対応を直前ターンから続けた場合、以下を結果差分へ追加する。

| 対応 | 2連続 | 3連続以上 |
| --- | --- | --- |
| `catch` | 追加負荷 +1 | 追加負荷 +2、流れ -1 |
| `arrange` | 成功時の負荷 -1 を無効化 | 負荷 +1、評判 -1 |
| `wait` | 成功時の負荷 -1 を無効化 | 負荷 +1、流れ -1 |
| `cut` | 信頼 -2 | 評判 -1、流れ -1、信頼 -2、負荷 +1 |

対応カードの副作用欄にも、`連続使用: 負荷+1`、`連続使用: 負荷回復なし`、`連続使用: 信頼-2` のように事前表示する。結果画面では連続使用補正が発生した理由を内訳に表示する。

### 追加ペナルティ

| 条件 | 効果 |
| --- | --- |
| 結果確定前の裏方負荷が4以上、かつランクが名場面ではない | 流れ差分 -1 |

### 転換 + 切るの下振れ軽減

`prepareTransition` を選び、かつ対応で `cut` を選んだ場合、転換系の出来事を強く閉じ、下振れを抑える。単発では強いが、連続使用では信頼と流れを大きく削る。

| 条件 | 効果 |
| --- | --- |
| 準備が活きた | スコア+2。崩れを閉じるだけでなく、次へ渡す場面の切れ味になる |
| 準備が一部活きた | スコア+1。読みの一部を使い、崩れを小さく閉じる |
| `prepareTransition + cut` | スコア+2。高負荷なら+3 |
| `accident` 相当、かつ準備が活きたまたは一部活きた | `fray` へ軽減する |
| `fray` 相当、かつ準備が活きた | `smallSuccess` へ軽減する |
| ほころび生成判定 | 通常より発生率を下げる |
| 負荷増加 | +1分を抑制する |

この組み合わせは常用する高得点手ではなく、崩れを小さく閉じて次へ送る判断として扱う。

## ほころび

ほころびは高負荷時に次ターン以降へ持ち越される舞台裏の未解決トラブル。画面上では `舞台裏のほころび` として表示する。単なるペナルティではなく、系統に合う対応を選ぶと場面化の材料になり、合わない対応を選ぶと流れや負荷へ軽い悪影響を残す。

### 内部strain

裏方負荷の数値は `backstageLoad` 1本で管理する。一方で、ほころびの系統を直近行動だけで決めないため、内部的に `loadStrain` を持つ。

| strain | 対応 |
| --- | --- |
| `light` | `catch` による負荷 |
| `sound` | `wait` による負荷 |
| `stageManagement` | `arrange` による負荷 |
| `props` | `cut` による負荷 |

負荷差分が正なら、選んだ対応の系統strainが増える。負荷差分が負なら、その対応の系統strainが抜ける。各日ソワレ後の回復では全系統strainが少し抜ける。HUDでは4系統のstrain値そのものは表示せず、負荷3以上で `乱れそう: 音響まわり` のように最も強い系統だけを短く表示する。

### 生成条件

結果確定後の負荷見込み `nextLoad = clamp(backstageLoad + deltaLoad)` を使う。

1. `nextLoad` が4未満なら生成しない。
2. 結果ランクが名場面なら生成しない。
3. 結果ランクが `fray` または `accident` なら必ず生成する。
4. それ以外のランクでは48%の確率で生成する。

ほころびの系統は、更新後の `loadStrain` を重みとして選ぶ。strainがない場合のみ、`preview.loadBias`、現在の `state.loadBias`、`stageManagement` の順でフォールバックする。

### ほころび系統とタイトル

| 系統 | 表示名 | 候補タイトル |
| --- | --- | --- |
| `light` | 光 | 照明が一歩遅れた / スポットの輪郭が揺れた |
| `sound` | 音 | 音が感情を先取りしすぎた / 残響が少し長く残った |
| `stageManagement` | 進行 | 転換が急ぎすぎた / 進行の合図が一拍ずれた |
| `props` | 道具 | 小道具の受け渡しが遅れた / 小道具の準備が詰まった |

現実装では対応から直接発生する負荷系統は `light`、`sound`、`stageManagement`、`props` の4種類。`cut` は場面を閉じて転換と道具を次へ送る判断として、`props` 系統の負荷を発生させる。

### 回収

ほころびが残っている状態で、系統に合う対応を選ぶと結果スコアに+1される。強く合う対応で、かつ直前と違う対応で拾った場合は、追加で+1され、結果ランクを1段階底上げし、成功時は評判・流れ・信頼にも+1される。強く合う対応で事故相当になった場合は、ほころびへ軽減することがある。結果ランクが事故以外なら、場面タイトル生成時にほころび系統を「回収された系統」として優先的に使う。

合わない対応を選んだ場合、結果スコア加点はない。結果差分に流れ-1を追加し、負荷4以上なら負荷+1も追加する。新しいほころびが発生しなかった場合、既存のほころびは持ち越される。

## 役者状態更新

結果確定後、次ターンに入る前に役者状態を更新する。乱数seedは `seed:state:totalTurn:logs.length`。

各役者について以下を行う。

### 疲労更新

- 26%の確率で疲労-1。
- 疲労は0-3に丸める。
- 裏方負荷は役者の疲労を直接増やさない。負荷は舞台裏側のリスクとして扱い、役者ステータスとは切り離す。

### 個別信頼更新

- 焦点役者の直近ログの信頼差分が+2以上なら個別信頼+2。
- 焦点役者の直近ログの信頼差分が+1なら個別信頼+1。
- 焦点役者の直近ログの信頼差分が負なら個別信頼-1。
- 個別信頼は0未満にならない。
- 個別信頼が3以上ある役者は、得意な対応を選んだ時に結果スコア+1され、事故相当をほころびへ軽減する。表示上は3以上で `阿吽の呼吸`、5以上で `以心伝心` のパッシブ効果として扱い、補正がない時は表示しない。

### 状態更新

- 焦点役者は72%の確率で状態を変更する。
- 焦点以外は34%の確率で状態を変更する。
- 状態変更時は5状態からランダムに選ぶ。
- 疲労が3以上なら、選ばれた状態に関係なく `fatigued` になる。

| `review` | 公演レビュー |
| `highlights` | 最大5件のハイライト |
| `insight.totalScore` | 公演点。通常の終演ランク、履歴、同seed比較に使う |
| `insight.discoveryScore` | 発見点。図鑑/称号などの初回解放報酬で、同seed比較には使わない |
| `insight.buildStyle` | 選択傾向から集計した公演ビルド、型レベル、進捗 |
| `insight.unlockedAchievements` | この公演で新たに解放された称号 |
| `insight.sceneCollectionCount` | この公演ログから図鑑対象になる場面数 |
| `insight.performanceBadges` | 準備的中、低負荷、千秋楽場面化など、その公演固有の短い達成バッジ |

### 公演ランク

終演画面の総合評価点は以下で計算する。

```txt
totalScore = sceneScore * 2 + flowScore + trustScore - backstageLoad * 2 - highLoadPenalty
highLoadPenalty = backstageLoad >= 4 ? 3 : 0
```

評判は大きく評価するが、裏方負荷が高い公演は終演ランクで重く減点する。

### 公演タイトル

以下の優先順で決まる。

| 条件 | タイトル |
| --- | --- |
| 名場面が2回以上、かつ `catch` が2回以上 | 予定外を評判に変えた公演 |
| `wait` が2回以上、かつ信頼スコアが4以上 | 余韻が残る千秋楽 |
| `catch` が3回以上 | 熱量で押し切った公演 |
| `arrange` が3回以上、かつ段取りスコアが評判スコア以上 | 段取りが支えた公演 |
| 流れスコアが0未満、または信頼スコアが0未満 | 荒れたが忘れがたい三日間 |
| 上記以外 | 三日間を支えきった公演 |

### 公演レビュー

レビューは3つの文を連結して作る。

1. `catch` 回数が `wait` 回数以上なら、予定外を拾う判断が熱を与えた旨。そうでなければ、待つ判断が余韻を残した旨。
2. `arrange` が3回以上なら、乱れを整えて流れを守った旨。そうでなければ、揺れが本番らしさになった旨。
3. `masterpiece` または `scene` のログがあれば、終盤側から最大3件の場面タイトルを挙げる。なければ、派手な名場面は少ないが舞台は途切れなかった旨。

### ハイライト

ハイライトは以下の手順で最大5件選ぶ。

1. 全ログを結果ランク順に並べる。ランク順は `masterpiece` > `scene` > `smallSuccess` > `fray` > `accident`。
2. 同ランクでは評判差分が大きいものを優先する。
3. 上位5件を取る。
4. 表示順は総ターン昇順に戻す。

## 履歴保存

終演時、公演結果は `localStorage` に保存される。

- キー: `honban.performance.history.v1`
- 保存件数: 最新8件まで。
- 保存形式: `PerformanceResult[]` のJSON。
- 読み込み失敗時は空配列として扱う。

タイトル画面では履歴を表示し、履歴項目を押すと同じ巡り合わせで再演できる。履歴には公演ランク、評判、段取り、座組信頼、準備ヒット数、名場面数を表示する。同じ巡り合わせの古い履歴がある場合は前回比を表示し、履歴内の最高評価、評判最高、最少負荷も短く示す。内部的には同じseedの履歴として照合する。

履歴には公演バッジも表示する。バッジは保存データ内の `insight.performanceBadges` に含まれる派生情報であり、性能には影響しない。タイトル画面ではバッジで履歴を絞り込み、該当するseedを再演できる。

## 次の公演目標

タイトル画面と終演画面では、複数の導線を横並びにせず、状況から1つの主目標を推薦する。

優先順位は以下。

1. 同seedで改善できるターンがあり、差し替え候補が出せる場合は再演。
2. 当日の日替わり挑戦が未プレイなら日替わり。
3. 次ランクまで近いseedがあれば再演。
4. 図鑑の未開放ヒントが残っていれば新seed。
5. 上記以外は別seedで新しい公演の型を探す。

同seed再演の具体候補は `src/game/replayAnalysis.ts` で算出する。対象公演のログをseedから再構築し、改善候補ターンだけ準備/対応を差し替え、その後のターンは元ログの選択を再利用して終演までシミュレーションする。結果が改善する候補があれば、`締め -> 整える / 総合+6` のように次の公演目標へ表示する。

## ローグライト進捗保存

公演履歴とは別に、場面図鑑、称号、日替わり自己ベストを `localStorage` に保存する。いずれもゲーム進行を有利にする永続強化ではなく、発見、記録、再挑戦動機を作るための保存である。

| キー | 内容 | 実装 |
| --- | --- | --- |
| `honban.collection.v1` | 解放済み場面、最高ランク、初回seed、称号 | `src/game/rogueliteProgress.ts` |
| `honban.daily.best.v1` | 日付ごとの日替わり挑戦自己ベスト | `src/game/rogueliteProgress.ts` |

読み込み失敗時は空の状態として扱う。今後保存形式を拡張する場合も、壊れたJSONや古いJSONでタイトル画面が落ちないようにする。

### 場面図鑑

場面図鑑は `sceneTitle` を中心に収集する。解放済み場面は最高結果ランク、初回seed、最後に出したseedを持つ。未解放の代表場面はタイトル画面にヒントとして表示する。

新しい場面タイトル、役者イベント、対応を追加した場合は、必要に応じて `SCENE_HINT_CATALOG` と解放条件を同時に更新する。

### 称号

称号はプレイヤーの達成や選択傾向を記録する。例として、拾う成功の累積、待つ千秋楽、低負荷終演、全対応使用、型レベル達成などを扱う。

称号はスコアや選択肢性能に影響しない。スコアアタック公平性を守るため、今後も性能上昇型の永続解放は追加しない。

### 公演ビルド

対応選択から `heat`、`breath`、`control`、`closure` の型を集計する。型の進捗は結果画面と終演画面に表示し、一定レベル以上では公演点にボーナスを加える。

現行の型レベルは `src/game/rogueliteProgress.ts` のしきい値で決まる。採点への反映は `src/game/scoreRules.ts` が担当する。しきい値やボーナスを変更した場合は、`npm run balance:report -- --samples=48` で主要戦略の偏りを確認する。

## seedと乱数

乱数は文字列seedから決定的に生成される。

- seed生成: `honban-${Date.now().toString(36)}-${random}`。
- 乱数実装: `xmur3` による文字列ハッシュと、`mulberry32` 系の数値更新。
- 同じseedで開始すると、同じ選択をした限り同じ焦点役者、イベント、タイトル、状態変化が再現される。

用途ごとにseed文字列を分けている。

| 用途 | seed形式 |
| --- | --- |
| 焦点役者 | `seed:focus:totalTurn` |
| 役者イベント | `seed:event:totalTurn:focusId` |
| 場面タイトル | `seed:title:totalTurn:score` |
| ほころび | `seed:fray:totalTurn:nextLoad` |
| 役者状態更新 | `seed:state:totalTurn:logs.length` |

### 今日の巡り合わせ

日替わり挑戦は端末ローカル日付から次のseedを生成する。

```txt
honban-daily-YYYY-MM-DD
```

日替わりseedでは `src/game/dailyRun.ts` の修飾子が1つ選ばれ、役者イベント重みと負荷揺れに軽い偏りを与える。通常seedにはこの補正を適用しない。

現行の修飾子は以下。

| 修飾子 | 主な影響 |
| --- | --- |
| `roughOpening` | 序盤の揺れを強める |
| `juniorHeat` | 若手の熱/前進/アドリブを出やすくする |
| `leadSilence` | 主役の沈黙/退場遅れを出やすくする |
| `skilledDrift` | 技巧派の位置/群像の揺れを出やすくする |
| `heavyTransition` | 道具/進行系の負荷を重くする |
| `hotAudience` | 評判上昇と負荷上昇を強める |

## 画面仕様

### タイトル画面

表示内容。

- タイトル、キャッチコピー。
- 役者3人のシルエット。
- `はじめる` ボタン。
- `遊び方` ボタン。押すたびに説明パネルを開閉する。
- 次の公演目標。履歴、図鑑、日替わり自己ベストから主CTAを1つ推薦する。
- 最近の公演履歴。履歴がない場合は未記録文言を表示する。
- 公演バッジの履歴フィルタ。選んだバッジを持つ履歴だけに絞り込める。

### ゲーム画面

常時表示。

- ヘッダー: 現在のターン数、日、マチネ/ソワレ、公演の色、`公演を終了する` ボタン。
- スコアレール: 評判、段取り、信頼、裏方負荷5灯。
- フェーズ表示: `準備`、`対応`、`場面` の進行ランプ。
- 負荷3以上でほころびが未発生の場合、`乱れそう: 音響まわり` のように最も強い内部strainだけを表示する。
- ほころびが残っている場合、`舞台裏のほころび` と系統、タイトルを表示する。

`prep` では注目役者カードを表示する。

- 役割バッジ。注目役者はカード左上に `今回の注目役者` と表示する。
- 役者名。
- 役者タイプの説明。
- 状態ブロック。`状態`、状態名、状態の意味を表示する。
- 発生しやすい兆候上位3件。
- 他の役者は `控え` または `次の注目候補` バッジと、状態、発動中のパッシブ効果、次ターン注意を短く表示する。
- 準備カード、本番前メモ、`この準備で本番へ` CTAを表示する。
- 準備カードを選ぶと、本番前メモに反映され、選択中カードには `メモに反映` を表示する。
- 本番前メモの承認欄は最初 `未承認`。CTA押下後にメモ紙上で `承認済` へ変わる。

兆候上位3件はイベント重みを降順に並べる。表示強度は以下。

- 最上位の重みとの差が2以内なら `高`。
- 差が8以内なら `中`。
- それ以外は `低`。

`response` では注目役者カードに確定イベントを表示し、対応選択カードを表示する。確定イベントはこのフェーズの主役なので、`本番で起きた` と出来事名を大きく見せ、役者名や状態よりも出来事を優先して読ませる。

`result` では役者カードを非表示にし、結果プレビューカードを表示する。

### 結果プレビュー

表示内容。

- 結果ランク。
- 注目役者 / イベント / 対応。
- `準備が活きた` / `準備が一部活きた` / `別の備えだった`。
- 仕上がりと星。
- 準備回収テキスト。
- 場面タイトル。
- フレーバーテキスト。
- 評判、流れ、信頼、負荷の差分。
- 決め手、代償、申し送り、客席反応。
- 次の判断メモ。
- 主な理由。

終演画面では次の公演目標を表示する。低結果、準備ミス、高負荷などの改善しやすいターンがあれば、その公演回を同じ巡り合わせでの再演の詰めどころとして提示する。差し替えシミュレーションで改善候補が見つかった場合は、具体的な準備/対応と見込み差分を表示する。
- `この結果で次の回へ` / `この結果で公演報告へ` ボタン。

### 終演画面

表示内容。

- 終演ラベル。
- 公演タイトル。
- 公演レビュー。
- 最終スコア: 評判、流れ、信頼、負荷。
- 公演バッジ。
- 次の公演目標。
- ハイライト一覧。
- 観客アンケート、劇評、判断傾向、タイムラインは折りたたみの公演記録へまとめる。
- `別の公演を開ける`、`同じ巡り合わせでやり直す`、`タイトルへ` ボタン。

## 実装上の補足

- バックエンド、外部API、ランタイム画像生成は使っていない。
- 依存はReact、React DOM、Vite、TypeScript系のみ。
- ロジックテストは `tests/` 配下にあり、`npm run test:logic` で Node.js 標準テストランナーにより実行する。
- 結果スコアのルール本体は `src/game/scoreRules.ts` が担う。互換用の公開口として `src/game/scoring.ts` を残している。
- 対応見込みの入口は `src/game/responseInsight.ts`、結果プレビューの入口は `src/game/resultPreview.ts` に分離している。
- 日程計算は `src/game/turnCalendar.ts`、終演後のレビュー・観客アンケート・メディア評は `src/game/performanceReport.ts` に分離している。
- 同seed再演の差し替え候補は `src/game/replayAnalysis.ts` で算出する。
- ドメイン型、表示派生型、終演レポート型は `src/game/domainTypes.ts`、`src/game/viewTypes.ts`、`src/game/reportTypes.ts` に分離し、`src/game/types.ts` から再エクスポートしている。
- ターン確定処理は `src/game/gameReducer.ts` の `commitResult` に集約し、履歴保存の副作用は `src/app/usePerformanceHistory.ts` で行う。
- `props` 系統のほころびは、`cut` を選んだ結果として通常の対応から直接発生しうる。
- 役者ごとの個別信頼は、準備画面のパッシブ効果表示と、得意対応を選んだ時の下振れ軽減・スコア補正に使われる。
