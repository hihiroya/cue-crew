## ゲーム概要

- タイトル: `本番中 x 舞台裏`
- タイトルコピー: `兆候を読み 予定外を名場面へ`
- ジャンル: 1人用の舞台裏マネジメントゲーム
- プレイヤーの役割: 本番中の裏方。役者へ直接命令するのではなく、兆候を読み、本番前の準備を決め、予定外の出来事へ対応する。
- 1公演の長さ: 全3日。各日マチネ/ソワレの2回で合計6ターン。
- 勝敗条件: 明示的なゲームオーバーはない。6ターン終了時の公演結果、レビュー、ハイライト、ランク、記録更新によって評価される。
- 永続強化: なし。履歴、図鑑、称号、日替わり自己ベストは記録と再挑戦導線であり、スコアアタック公平性に影響しない。

## 早見表

### 1ターンの流れ

1. 焦点役者と兆候上位3件を確認する。
2. 準備を1つ選ぶ。
3. 選んだ準備を本番前メモへ反映し、承認して本番へ進む。
4. 役者イベントが確定する。
5. 対応を1つ選ぶ。
6. 結果プレビューでランク、差分、理由、場面タイトルを確認する。
7. 結果を確定し、スコア、ログ、負荷、ほころび、役者状態、次ターンを更新する。

### プレイヤーが選ぶもの

| フェーズ | 選択肢 | 数 | 結果への主な影響 |
| --- | --- | ---: | --- |
| 準備 | 注視、余白、締め、転換 | 4 | 準備品質、結果上限、対応との追加補正、負荷悪化抑制 |
| 対応 | 拾う、整える、待つ、切る | 4 | イベント相性、スコア、差分、負荷系統、ほころび回収 |

### 管理スコア

| スコア | 初期値 | 範囲 | 意味 |
| --- | ---: | --- | --- |
| 評判 `sceneScore` | 0 | 制限なし | 場面化、名場面化の積み上げ。終演評価で重く扱う。 |
| 段取り `flowScore` | 0 | 制限なし | 公演全体の進行の安定度。 |
| 一体感 `trustScore` | 0 | 制限なし | 公演全体の息の合い方。終盤の伸びやすさ、待つ判断の強さに関わる。役者個別の信頼とは別管理。 |
| 裏方負荷 `backstageLoad` | 0 | 0-5 | 高いほど結果判定や終演評価に不利。`clampLoad` で丸める。 |

## 公演構造

### 日程

| 日 | 名前 | 役割 | ターン |
| --- | --- | --- | --- |
| 1 | 1日目 | 土台作り。準備と一体感が公演の色を作る | 1-2 |
| 2 | 2日目 | 揺らぎ。公演の色を活かして攻めるか、負荷を整える | 3-4 |
| 3 | 3日目 | 千秋楽。残った一体感と負荷を回収して閉じる | 5-6 |

日と公演回は総ターンから計算する。

- `act = ceil(totalTurn / 2)`。内部名は既存互換のため `act` のままだが、UIでは「日」として扱う。
- `turnInAct = 1` はマチネ、`turnInAct = 2` はソワレ。
- 各日ソワレの結果確定後、裏方負荷が1軽減される。

| 公演回 | 役割 | 表示方針 |
| --- | --- | --- |
| マチネ | 昼公演。見極めと調整が効きやすい | 次の回への申し送りと主な理由を短く表示する。 |
| ソワレ | 夜公演。評判は伸びるが、負荷も残りやすい | その日の評判、公演の色、負荷への影響を厚めに表示する。 |

### 公演の色

1日目ソワレ、つまり2ターン目の結果確定時から `determinePerformanceStyle` で公演の色を判定する。最上位の色と2位の差が4点以上なら確定し、差が薄い場合は未確定のまま次ターン以降に持ち越す。色は2日目以降の判断傾向に補正を与える。

| 色 | strength | 決まりやすい傾向 | 効果 |
| --- | --- | --- | --- |
| 熱量 `heat` | 拾う | `catch`、評判差分が高い | `catch` にスコア+1。成功時、評判+1、負荷+1。 |
| 余韻 `breath` | 待つ | `wait`、一体感差分が高い | `wait` にスコア+1。成功時、一体感+1。 |
| 段取り `control` | 整える | `arrange`、段取り上昇、負荷低下 | `arrange` にスコア+1。成功時、負荷-1。ほころび発生を抑えやすい。 |
| 収束 `closure` | 切る | `cut`、負荷を増やさず閉じる | `cut` にスコア+1。負荷悪化を抑えやすいが、一体感は伸びにくい。 |

色の決定スコアは、対応ごとの基礎点3点に加え、各ログの差分を足して最も高い色を採用する。ただし差分ボーナスはその色の strength に合う対応を選んだログだけに入る。

| 色 | 追加される進捗 |
| --- | --- |
| 熱量 | 評判のプラス差分 |
| 余韻 | 一体感のプラス差分 |
| 段取り | 段取りのプラス差分、負荷低下 |
| 収束 | 段取り低下、負荷を増やさない結果 |

## ゲーム状態

`GameState.status` は以下の5種類。

| 状態 | 内容 |
| --- | --- |
| `title` | タイトル画面。新規開始、履歴から同seedで再演、遊び方表示ができる。 |
| `prep` | 準備フェーズ。焦点役者、状態、兆候、準備カード、本番前メモを表示する。 |
| `response` | 対応フェーズ。準備確定後に役者イベントが確定し、4つの対応を選ぶ。 |
| `result` | 結果プレビュー。選択した対応の結果、差分、理由、次メモを表示する。 |
| `finished` | 終演画面。最終結果、レビュー、ハイライト、再演操作を表示する。 |

### アクション

| アクション | 有効状態 | 効果 |
| --- | --- | --- |
| `START` | 任意 | 新しい公演を開始する。seed指定時はそのseedで開始する。 |
| `RESET_TO_TITLE` | 任意 | タイトル状態へ戻る。 |
| `SELECT_PREP` | `prep` | 準備を保存し、役者イベントを解決して `response` へ進む。 |
| `SELECT_RESPONSE` | `response` または `result` | 対応を保存し、`result` へ進む。 |
| `COMMIT_RESULT` | `result` | 結果を確定し、スコア、ログ、役者状態、次ターンを更新する。 |

### 初期状態

`createInitialGame(seed)` は以下を作る。

- `totalTurn = 1`
- `act = 1`
- `turnInAct = 1`
- スコアはすべて0
- `performanceStyle = null`
- `backstageLoad = 0`
- `loadBias = null`
- `loadStrain = { light: 0, sound: 0, stageManagement: 0, props: 0 }`
- 焦点役者は `pickFocusActor(seed, 1)` で決定
- `status = prep`

## 役者

役者は3人。

| ID | 表示名 | 初期状態 | 初期信頼 | 初期疲労 | 初期役割 |
| --- | --- | --- | ---: | ---: | --- |
| `lead` | 主役 | 沈思 | 0 | 0 | 焦点 |
| `junior` | 若手 | 高揚 | 0 | 0 | 兆候 |
| `skilled` | 技巧派 | 没入 | 0 | 0 | 待機 |

| 役者 | 特徴 |
| --- | --- |
| 主役 | 間の型。沈黙・退場の余韻が多い。待つと伸びやすい。 |
| 若手 | 勢い型。前へ出る・アドリブ・熱が乗るが多い。拾うと伸びやすい。 |
| 技巧派 | 制御型。立ち位置・群像の乱れが多い。整えると伸びやすい。 |

### 役者が持つ値

| 値 | 内容 |
| --- | --- |
| `state` | 現在の心理・身体状態。 |
| `trust` | 個別役者の信頼値。3以上で得意対応のパッシブ補正が出る。 |
| `fatigue` | 0-3の疲労値。3以上になると状態が疲労へ固定される。 |
| `currentRole` | 画面上の役割。`focus`、`sign`、`waiting`。 |

### 役者状態

| ID | 表示名 |
| --- | --- |
| `elated` | 高揚 |
| `contemplative` | 沈思 |
| `anxious` | 不安 |
| `immersed` | 没入 |
| `fatigued` | 疲労 |

状態変化はseed付き乱数だが、完全な一様ランダムではない。直前ログの結果に応じて状態ごとの重みを変える。

| 直前結果 | 上がりやすい状態 |
| --- | --- |
| 名場面、または評判差分が大きい | 高揚、没入 |
| `catch` 成功 | 高揚 |
| `wait` 成功、かつ一体感が伸びた | 沈思、没入 |
| `arrange` 成功、かつ段取りが伸びた | 沈思、没入 |
| `cut` | 沈思。さらに一体感が下がった場合は不安 |
| ほころび、事故、段取り低下 | 不安 |
| 負荷+2以上、または裏方負荷4以上 | 疲労、不安 |

焦点以外の役者も、全体の一体感、段取り低下、負荷の重さから軽く影響を受ける。疲労が3になると状態は `fatigued` に固定される。

## 焦点役者

各ターンの焦点役者はseed付き乱数で決まる。初日だけは開始パターンの固定感を避けるため、通常の優先順とは別に扱う。

初日の判定手順。

1. 1ターン目は `seed:focus:opening` から乱数を作り、若手40%、主役30%、技巧派30%で選ぶ。
2. 2ターン目は `seed:focus:2` から乱数を作り、1ターン目とは別の役者を選ぶ。
3. 2ターン目の候補比率は通常の優先順リストから1ターン目の役者を除いた比率を使う。

3ターン目以降の優先順は以下。

| ターン | 優先焦点 |
| ---: | --- |
| 3 | 技巧派 |
| 4 | 若手 |
| 5 | 主役 |
| 6 | 技巧派 |
| 7 | 若手 |
| 8 | 技巧派 |
| 9 | 主役 |
| 10 | 若手 |

3ターン目以降の判定手順。

1. `seed:focus:totalTurn` から乱数を作る。
2. 72%の確率で優先焦点を採用する。
3. 28%の確率で上記の優先順リスト全体からランダムに選ぶ。

焦点以外の役者の表示役割は以下。

- 焦点役者は `focus`。
- 焦点以外で、若手または状態が不安の役者は `sign`。
- それ以外は `waiting`。

## 役者イベント

イベントは8種類。

| ID | 表示名 | 説明 |
| --- | --- | --- |
| `stepForward` | 前へ出る | 予定より半歩前に出て、客席の視線をさらいかけている。 |
| `adlib` | アドリブ | 台詞の隙間に、台本にはない一言が落ちた。 |
| `heatUp` | 熱が乗る | 言葉と身体に熱が宿り、場面が大きく膨らみ始めた。 |
| `silence` | 沈黙する | 台詞が止まり、舞台に息を呑むような間が生まれた。 |
| `positionShift` | 立ち位置がズレる | 立ち位置が少しズレ、照明と視線の軸が変わった。 |
| `tempoRush` | テンポが走る | 呼吸より先に台詞が走り、舞台全体の拍が揺れている。 |
| `delayedExit` | 退場が遅れる | 退場の一歩が遅れ、余韻と進行の境目で揺れている。 |
| `ensembleWaver` | 群像が揺れる | 周囲の動きがわずかに乱れ、群像の輪郭が揺らいだ。 |

### イベント抽選

1. 焦点役者の種類別重みと状態別重みを足す。
2. 焦点役者の疲労が2以上の場合、`positionShift`、`tempoRush`、`ensembleWaver` にそれぞれ+3する。
3. 日替わりseedの場合は日替わり補正を加える。
4. `seed:event:totalTurn:focusId` から乱数を作る。
5. `0.14 + backstageLoad * 0.04 + dailyVolatilityBonus` の確率で本番の揺れが入り、重みを平準化した抽選を行う。
6. 上記以外は合計重みによる加重抽選を行う。

準備は役者イベントの抽選seedには含まれない。準備は「起きた出来事に備えられていたか」と「本番中の対応へどうつながるか」の判定に使う。

## 準備

準備フェーズでは4種類から1つ選ぶ。準備はイベント抽選には影響しないが、準備品質、結果上限、対応との追加補正、負荷悪化抑制に影響する。

| ID | 表示名 | 活きやすい対応 | 備える出来事 | 外れた時の受け方 |
| --- | --- | --- | --- | --- |
| `watch` | 注視 | 拾う | 前へ出る、アドリブ、熱が乗る | 整えるで安全寄りに処理できる |
| `makeSpace` | 余白 | 待つ | 沈黙する、退場が遅れる、熱が乗る | 熱が来たら拾うと伸びる |
| `tightenFlow` | 締め | 整える | 立ち位置がズレる、テンポが走る、群像が揺れる | 高負荷なら切って守る手もある |
| `prepareTransition` | 転換 | 切る | テンポが走る、退場が遅れる、群像が揺れる、立ち位置がズレる | 低負荷なら整える・待つで場面化を狙える |

準備カードには、準備名、備える兆候の灯り、活きやすい対応アイコン、備える揺れカテゴリ、メモ反映状態を表示する。対応アイコンは「次の行動が固定された」という意味ではなく、「この準備は本番でその対応を選ぶと活かしやすい」という相性を示す。

選択中の準備は本番前メモへ反映される。メモには、見えている兆候、同じ準備で拾える出来事、備えどおりに来た場合、外れた場合、承認欄を表示する。CTA `この準備で本番へ` を押すと、メモ紙上の承認欄が `未承認` から `承認済` へ変わり、対応フェーズへ進む。

### 準備品質

| 品質 | 条件 | スコア補正 | 結果上限 |
| --- | --- | ---: | --- |
| `hit` 活きた | 発生イベントが選択準備の対応イベントに含まれる | +1 | 制限なし |
| `partial` 一部活きた | 発生イベントとは違うが、見えている兆候上位3件には備えていた | 0 | 場面化まで。準備の活きやすい対応から外れるとほころびまで |
| `miss` 別の備えだった | 発生イベントにも見えている兆候にも備えていない | -1 | 小さな成功まで |

`miss` かつ準備の活きやすい対応から外れ、直前の `cut` 収束もない場合は、結果上限がほころびまで下がる。直前に `cut` で場面を閉じていた場合、`miss` でも上限は場面化まで広がる。`arrange` は安定手として扱うため、準備が活きても、焦点役者が技巧派、不安/疲労、またはほころび回収などの強い理由がない場合は名場面に届かず場面化までに留まる。

### 準備と対応の追加関係

| 条件 | 効果 |
| --- | --- |
| 準備の活きやすい対応を選ぶ | 準備品質に応じた追加補正。`hit` は基本+1。 |
| `prepareTransition + cut` かつ `hit` | 追加補正+2。事故やほころびの軽減も持つ。 |
| `prepareTransition + cut` かつ `partial` | 追加補正+1。 |
| `watch + arrange` | 代替筋として安全寄りに処理できる。 |
| `makeSpace + catch` かつイベントが `heatUp` | 熱を拾う代替筋として伸びる。 |
| `tightenFlow + cut` | 高負荷時の守り筋になる。 |
| `prepareTransition + wait/arrange` | 低負荷時に場面化を狙う代替筋になる。 |
| 発生イベントとの相性が正の対応 | 備え外でも有効筋として扱う。 |
| 準備の活きやすい対応から外れる | `partial` は -1、`miss` は -2。画面では「備えから逃げた」と短く表示する。 |

## 対応

準備確定後、役者イベントに対して4種類から1つ選ぶ。

| ID | 表示名 | 説明 | 負荷系統 |
| --- | --- | --- | --- |
| `catch` | 拾う | 予定外の行動を見せ場に変える | 光 |
| `arrange` | 整える | 乱れを舞台全体の段取りに戻す | 進行 |
| `wait` | 待つ | 間や余韻を信じて急かさない | 音 |
| `cut` | 切る | 場面を閉じ、転換と道具を次へ送る | 道具 |

対応は以下に影響する。

- イベント相性
- 役者タイプ補正
- 役者状態補正
- 準備との関係
- 公演日/公演回補正
- 公演の色補正
- 公演の色の深まり補正
- 役者信頼補正

## ほころび

ほころびは、裏方負荷と負荷系統strainから発生する舞台裏の残り火である。ゲーム上は `pendingFrayEvent` として1件だけ持つ。

### 負荷系統

| 系統 | 表示 | 対応 |
| --- | --- | --- |
| `light` | 光 | 拾う |
| `sound` | 音 | 待つ |
| `stageManagement` | 進行 | 整える |
| `props` | 道具 | 切る |

`loadStrain` は各系統0-5で管理する。結果の負荷差分が正なら対応系統に加算、負なら同系統から減算する。日またぎ軽減時は全系統を1ずつ下げる。

### 発生条件

結果確定時、`maybeCreateFray` が新しいほころびを作る。

1. `nextLoad = clamp(backstageLoad + deltaLoad, 0, 5)` を見る。
2. `nextLoad < 4` または結果が名場面なら発生しない。
3. `prepareTransition + cut`、または公演の色が `control`、または公演の色が `closure` かつ `cut` の場合はガード扱いになる。
4. ガードなしなら、結果が `fray` / `accident` の時に発生候補。さらに48%のランダム発生がある。
5. ガードありなら、結果が `accident`、または `fray` かつ38%の時に発生候補。ランダム発生率は18%。
6. 系統は `loadStrain` の重みによる抽選で決まる。strainがない場合は直近の負荷系統をfallbackにする。

乱数seedは `seed:fray:totalTurn:nextLoad`。

### 回収

残っているほころびに対し、系統ごとの強い対応または合う対応を選ぶと回収できる。

| 回収状態 | 条件 | スコア | 差分 | ランク軽減 |
| --- | --- | ---: | --- | --- |
| `strong` | 系統ごとの強い対応 | +2 | 連続同対応でなければ成功時に評判+1、段取り+1、一体感+1。さらに強回収は評判+1。 | 事故をほころびへ。さらに強回収は結果レンジを1段上げる。 |
| `match` | 系統ごとの合う対応 | +1 | なし | なし |
| `miss` | 合わない対応 | 0 | 段取り-1。負荷4以上なら負荷+1 | なし。ほころびは持ち越す。 |
| `none` | ほころびなし | 0 | なし | なし |

新しいほころびが発生した場合は新規ほころびを保持する。新規発生がなく、既存ほころびへの対応が `miss` の場合は既存ほころびを持ち越す。それ以外は解消する。

### 予兆表示

負荷3以上でほころびが未発生の場合、画面上に `乱れそう: 音響まわり` のように最も強いstrain系統だけを表示する。ほころびが残っている場合は `舞台裏のほころび` と系統、タイトルを表示する。

## 役者状態更新

結果確定後、`advanceActorStates` で役者状態を更新する。

### 疲労

- 各役者は通常26%の確率で疲労が1下がる。直近ログの負荷差分が負なら42%に上がる。
- 直近ログが負荷+2以上、または事故の場合、疲労圧として+1される。
- 疲労は0-3に丸められる。
- 疲労が3以上なら、状態変更時に必ず `fatigued` になる。

### 役者信頼

焦点役者だけが直近ログの内容から役者信頼を更新する。一体感 `trustScore` の差分をそのまま転用せず、焦点役者にとって納得しやすい扱いだったかで判定する。

| 条件 | 役者信頼 |
| --- | ---: |
| 成功し、役者の得意対応を選んだ | +1 |

| 準備品質が `miss`、かつイベント相性が0以下 | -1 |
| ほころび | -1 |
| 事故 | -2 |

| 技巧派以外の熱量系イベントを `arrange` | -1 |
| 段取り系イベントを `wait` | -1 |

1ターンあたりの変化は -2 から +2 に丸める。役者信頼は0未満にならない。

| 対象 | 状態変更率 |
| --- | ---: |
| 焦点役者 | 76% |

状態変更時は5状態から加重抽選する。重みは直前ログの結果、差分、対応、裏方負荷、疲労で変わる。乱数seedは `seed:state:totalTurn:logs.length`。

## seedと乱数

乱数は文字列seedから決定的に生成される。

- seed生成: `honban-${Date.now().toString(36)}-${random}`
- 乱数実装: `xmur3` による文字列ハッシュと、`mulberry32` 系の数値更新
- 同じseedで開始すると、同じ選択をした限り同じ焦点役者、イベント、タイトル、状態変化が再現される

| 用途 | seed形式 |
| --- | --- |
| 焦点役者 | `seed:focus:totalTurn` |
| 役者イベント | `seed:event:totalTurn:focusId` |
| 場面タイトル | `seed:title:totalTurn:score` |
| ほころび | `seed:fray:totalTurn:nextLoad` |
| 役者状態更新 | `seed:state:totalTurn:logs.length` |

| --- | --- |
| ほころび回収あり | 舞台裏のほころびが意味を持ち、役者イベントが本番だけの輪郭を得たことを表現する。 |
| 名場面 | 裏方が見逃さず、予定外の揺れが客席まで届く名場面になったことを表現する。 |
| 場面化 | 役者状態が生んだイベントを対応で受け止め、危うさを残しつつ舞台の呼吸が途切れなかったことを表現する。 |
| 小さな成功 | 大きな見せ場には届かないが、丁寧に処理して次へ渡したことを表現する。 |
| ほころび | 舞台が追いつかず、次に拾える余白として揺れが残ったことを表現する。 |
| 事故 | 役者イベントが段取りから離れ、判断も一拍遅れたが、生の熱が残ったことを表現する。 |

### 準備回収テキスト

結果プレビューでは、準備がどれだけ本番の出来事に活きたかを表示し、詳細なスコア内訳ではなく主な理由を短く併記する。

| 条件 | トーン | ラベル |
| --- | --- | --- |
| 準備が活き、結果が `fray` / `accident` ではない | `matched` | 準備が活きた |
| 見えていた兆候の一部に備えていた | `thin` または `partial` | 準備が一部活きた |
| 上記以外 | `missed` | 別の備えだった |

## 終演結果

6ターン目、つまり3日目ソワレの結果確定後、状態は `finished` になる。`finishPerformance` は以下を含む `PerformanceResult` を作る。

| 項目 | 内容 |
| --- | --- |
| `seed` | 公演seed |
| `finishedAt` | ISO文字列の終了日時 |
| `sceneScore` | 最終評判スコア |
| `flowScore` | 最終段取りスコア |
| `trustScore` | 最終一体感スコア |
| `backstageLoad` | 最終裏方負荷 |
| `performanceStyle` | 1日目ソワレ後に決まった公演の色 |
| `title` | 公演タイトル |
| `review` | 公演レビュー |
| `reviewNotes` | レビュー構成文 |
| `insight` | 公演点、ランク、傾向、称号、バッジなどの派生分析 |
| `audienceSurvey` | 観客アンケート風の4指標 |
| `mediaReview` | 劇評風の星、見出し、引用 |
| `logs` | 全ターンログ |
| `highlights` | 最大5件のハイライト |

### 公演点とランク

終演画面の総合評価点は以下で計算する。

```txt
totalScore = sceneScore * 2 + flowScore + trustScore - backstageLoad * 2 - highLoadPenalty
highLoadPenalty = backstageLoad >= 4 ? 3 : 0
```

| ランク | しきい値 |
| --- | ---: |
| S+ | 52 |
| S | 44 |
| A | 34 |
| B | 24 |
| C | 14 |
| D | それ未満 |

評判は大きく評価するが、裏方負荷が高い公演は終演ランクで重く減点する。

### 公演タイトルとレビュー

公演タイトルとレビューは `src/content/ja/reportCopy.ts` の条件文言から作る。入力として使う主な値は以下。

- `catch` / `wait` / `arrange` / `cut` の回数
- 名場面回数
- 準備的中数と的中率
- ほころび/事故回数
- 公演の色
- 終盤の回収ログ
- 最終負荷

レビューは複数の短文を連結して作る。ほころび/事故が3回以上ある場合は追加のレビュー文を足す。

### ハイライト

ハイライトは以下の手順で最大5件選ぶ。

1. 全ログを結果ランク順に並べる。ランク順は `masterpiece` > `scene` > `smallSuccess` > `fray` > `accident`。
2. 同ランクでは評判差分が大きいものを優先する。
3. 上位5件を取る。
4. 表示順は総ターン昇順に戻す。

### 公演バッジ

終演ごとの短い達成表示。最大4件まで表示する。

| ID | 条件 | トーン |
| --- | --- | --- |
| `sharp-read` | 準備的中5回以上 | good |
| `double-masterpiece` | 名場面2回以上 | gold |
| `light-load` | 最終負荷1以下 | cool |
| `clean-run` | ほころび/事故0回、かつ6ログ以上 | good |
| `finale-scene` | 千秋楽ソワレが名場面または場面化 | gold |
| `style-max` | 公演の色の深まりが最大相当 | gold |
| `many-discoveries` | 発見点10以上 | cool |
| `all-cues` | 全対応を使い、場面化以上3回以上 | cool |
| `hot-backstage` | 最終負荷4以上 | risk |
| `steady-scenes` | 他バッジなし、場面化以上2回以上 | good |

## 履歴保存

終演時、公演結果は `localStorage` に保存される。

| キー | 内容 | 実装 |
| --- | --- | --- |
| `honban.performance.history.v1` | 最新8件の `PerformanceResult[]` | `gameReducer.ts` |

読み込み失敗時は空配列として扱う。古い保存データは `normalizePerformanceResult` で不足項目を補い、補正後のJSONを保存し直す。

タイトル画面では履歴を表示し、履歴項目を押すと同じ巡り合わせで再演できる。履歴には公演ランク、評判、段取り、一体感、準備ヒット数、名場面数、公演バッジ、同seed前回比などを表示する。

## ローグライト進捗保存

公演履歴とは別に、場面図鑑、称号、日替わり自己ベストを `localStorage` に保存する。これらはゲーム進行を有利にする永続強化ではなく、発見、記録、再挑戦動機を作るための保存である。

| キー | 内容 | 実装 |
| --- | --- | --- |
| `honban.collection.v1` | 解放済み場面、称号 | `rogueliteProgress.ts` |
| `honban.daily.best.v1` | 日付ごとの日替わり自己ベスト | `rogueliteProgress.ts` |

読み込み失敗時は空の状態として扱う。

### 場面図鑑

場面図鑑は `sceneTitle` を中心に収集する。IDは以下。

```txt
focusActorType:actorEventType:mainResponse:sceneTitle
```

解放済み場面は、タイトル、役者、イベント、対応、最高結果ランク、初回seed、初回日時を持つ。再度同じ場面が出た場合は最高結果ランクだけ更新する。

### 称号

称号はプレイヤーの達成や選択傾向を記録する。

| 称号ID | 条件 |
| --- | --- |
| `heat-catcher` | `catch` による名場面が2回以上 |
| `finale-breath` | 3日目に `wait` で場面化以上 |
| `flow-keeper` | `arrange` で場面化以上、かつ負荷低下 |
| `clean-blackout` | `cut` で事故以外 |
| `light-backstage` | 最終負荷1以下 |
| `read-the-room` | 準備的中5回以上 |
| `all-cue-run` | 全対応を使い、場面化以上3回以上 |
| `heat-finale` | 熱量の色が濃い状態で、かつ千秋楽ソワレが場面化以上 |
| `breath-finale` | 余韻の色が濃い状態で、かつ千秋楽ソワレが場面化以上 |
| `control-finale` | 段取りの色が濃い状態で、かつ千秋楽ソワレが場面化以上 |
| `closure-finale` | 収束の色が濃い状態で、かつ千秋楽ソワレが場面化以上 |

称号はスコアや選択肢性能に影響しない。

### 公演の色の深まり

対応選択から `heat`、`breath`、`control`、`closure` の色を集計する。進捗は結果画面と終演画面に表示し、一定以上では色strengthの対応にスコアボーナスを加える。内部的には `level` を持つが、画面では `Lv.1` などの数字ではなく、色の深まりとして表示する。

| 進捗 | 画面上の意味 |
| ---: | --- |
| 2以上 | 色の芽が出た |
| 5以上 | 色が濃く出ている |
| 9以上 | 色が公演を引っぱる |

### 発見点

`discoveryScore` は以下で計算する。

```txt
discoveryScore = sceneIds.length * 2 + unlockedAchievements.length * 5
```

発見点は図鑑/称号などの初回解放報酬であり、同seed比較や通常の終演ランクには使わない。

## 次の公演目標

タイトル画面と終演画面では、複数の導線を横並びにせず、状況から1つの主目標を推薦する。

### 終演後

優先順位は以下。

1. 同seedで改善できるターンがある、または差し替えシミュレーション候補があるなら同seed再演。
2. 当日の日替わり挑戦が未プレイなら日替わり。
3. 次ランクまで8点以内なら同seed再演。
4. 図鑑の未開放ヒントが残っていれば新seed。
5. 上記以外は選択傾向に応じて別seedを推薦。

### タイトル画面

優先順位は以下。

1. 履歴がなければ新規開始。
2. 履歴内に改善しやすい公演があれば再演。
3. 当日の日替わり挑戦が未プレイなら日替わり。
4. 次ランクまで8点以内の履歴があれば再演。
5. 図鑑の未開放ヒントが残っていれば新seed。
6. 上記以外は別seedで新しい公演の色を探す。

### 同seed再演の差し替え候補

`analyzeReplayImprovement` は以下の流れで候補を出す。

1. `mostImproveableTurn` で、ほころび/事故、準備ミス、負荷+2以上のターンから改善対象を選ぶ。
2. そのターンだけ全準備4種×全対応4種を試す。
3. それ以降のターンは元ログの選択を再利用し、終演までシミュレーションする。
4. 終演点が上がる、または対象ターンのランクが上がる候補だけ残す。
5. 終演点差分、対象ターンランク、負荷差分の順に並べ、最良候補を表示する。

## 画面仕様

### タイトル画面

表示内容。

- タイトル、キャッチコピー。
- 役者3人のシルエット。
- `はじめる` ボタン。
- `遊び方` ボタン。押すたびに説明パネルを開閉する。
- 今日の巡り合わせ。
- 次の公演目標。履歴、図鑑、日替わり自己ベストから主CTAを1つ推薦する。
- 最近の公演履歴。履歴がない場合は未記録文言を表示する。
- 公演バッジの履歴フィルタ。選んだバッジを持つ履歴だけに絞り込める。

### ゲーム画面共通

常時表示。

- ヘッダー: 現在のターン数、日、マチネ/ソワレ、公演の色、`公演を終了する` ボタン。
- スコアレール: 評判、段取り、一体感、裏方負荷5灯。
- フェーズ表示: `準備`、`対応`、`場面` の進行ランプ。
- 負荷3以上でほころびが未発生の場合、最も強い内部strainの予兆。
- ほころびが残っている場合、`舞台裏のほころび` と系統、タイトル。

### 準備画面

`prep` では注目役者カードを表示する。

- 役割バッジ。注目役者はカード左上に `今回の注目役者` と表示する。
- 役者名。
- 役者タイプの説明。
- 状態ブロック。`状態`、状態名、状態の意味を表示する。
- 発生しやすい兆候上位3件。
- 他の役者は `控え` または `次の注目候補` バッジと、状態、発動中のパッシブ効果、次ターン注意を短く表示する。
- 準備カード、本番前メモ、`この準備で本番へ` CTAを表示する。
- 準備カードを選ぶと、本番前メモに反映され、選択中カードには `メモに反映` を表示する。
- 本番前メモの承認欄は最初 `未承認`。CTA押下後にメモ紙上で `承認済` へ変わる。

### 対応画面

`response` では注目役者カードに確定イベントを表示し、対応選択カードを表示する。確定イベントはこのフェーズの主役なので、`本番で起きた` と出来事名を大きく見せ、役者名や状態よりも出来事を優先して読ませる。

表示内容。

- 確定した役者イベント。
- 対応カード4枚。
- 進行卓 HUD。
- キューバーと `このキューを出す` CTA。
- 準備との関係、成功幅、上振れ、下振れ、相性盤、副作用、ほころび回収、パッシブ効果。

候補カードのトーンは成功幅と危険警告に応じて変わる。`事故` は通常の見込みとしては表示せず、下振れが事故の恐れを含む時だけ `危険: 事故の恐れ` として表示する。

| 状態 | トーン |
| --- | --- |
| 名場面まで伸びる | best |
| 場面化まで届く | good |
| ほころびが絡む | thin |
| 事故の恐れ警告 | danger |

### 結果プレビュー

`result` では役者カードを非表示にし、結果プレビューカードを表示する。

表示内容。

- 結果ランク。
- 注目役者 / イベント / 対応。
- `準備が活きた` / `準備が一部活きた` / `別の備えだった`。
- 仕上がりと星。
- 準備回収テキスト。
- 場面タイトル。
- フレーバーテキスト。
- 評判、段取り、一体感、負荷の差分。
- 決め手、代償、申し送り、客席反応。
- 次の判断メモ。
- 主な理由。
- `この結果で次の回へ` / `この結果で公演報告へ` ボタン。

マチネはソワレへの影響を薄く、ソワレはその日の評判と型への影響を厚く表示する。千秋楽ソワレは `finale` 表示になる。

### 終演画面

表示内容。

- 終演ラベル。
- 公演タイトル。
- 公演レビュー。
- 最終スコア: 評判、段取り、一体感、負荷。
- 公演ランク、次ランクまでの点数。
- 公演バッジ。
- 次の公演目標。
- ハイライト一覧。
- 観客アンケート、劇評、判断傾向、タイムラインを折りたたみの公演記録へまとめる。
- `別の公演を開ける`、`同じ巡り合わせでやり直す`、`タイトルへ` ボタン。

## 保守チェックリスト

### 採点、イベント重み、日替わり補正、公演の色ボーナス、称号/図鑑条件を変更した場合

1. この文書の該当表を更新する。
2. `npm run test:logic` を実行する。
3. `npm run balance:report -- --samples=48` で主要戦略とイベント分布を確認する。
4. 変更がUI表示にも影響する場合は `npm run check:ui` または影響範囲の `check:ui:*` を実行する。

### UI仕様を変更した場合

1. この文書の「画面仕様」を更新する。
2. 通常は `npm run check:ui` を実行する。
3. 軽微なCSS/文言/余白修正では、必要に応じて `npm run check:ui:prep`、`npm run check:ui:response`、`npm run check:ui:result` など影響範囲だけ確認する。
4. スクリーンショット確認は通常使わない。大きなレイアウト改修、PR前に明示的に必要な場合、または指示がある場合だけ `npm run verify:ui:full` を使う。

### 保存形式を変更した場合

1. `PerformanceResult`、`CollectionState`、`DailyBest` の型と保存キーを確認する。
2. 壊れたJSON、古いJSON、空データでタイトル画面が落ちないことを確認する。
3. 必要に応じて `normalizePerformanceResult` の補完を更新する。

### 文言を変更した場合

1. `src/content/ja` の責務に合わせて追加する。
2. `npm run check:copy` を実行する。
3. 画面表示の密度が変わる場合は影響範囲のUIチェックを行う。

### 依存を追加・更新した場合

1. `package.json` と `package-lock.json` の差分を説明する。
2. 新しく入った direct dependency を報告する。
3. install script を持つ dependency を報告する。
4. 必要な依存は安全性を確認したうえで追加・更新してよい。

## 非目標

- バックエンド、外部API、ランタイム画像生成は使っていない。
- 永続強化による性能上昇は追加しない。
- 図鑑、称号、日替わり自己ベストは記録と再挑戦動機であり、通常公演の採点性能には影響しない。
